==================================================
About identity recognition
==================================================

.. identity-recognition-intro-start

Every event sent to an event stream can match to a known customer profile in real-time. Matching occurs when a value in the event matches a value for one or more linking keys configured for the event stream. Matching occurs before attributes are updated and before segments are evaluated.

Anonymous profiles are created when events do not match to a known profile. Anonymous profiles can eventually merge into a known profile when linking keys with values that match are discovered for the anonymous profile.

.. identity-recognition-intro-end


.. _identity-recognition-what-is:

What is identity recognition?
==================================================

.. TODO: This is, at best, a draft. There should be more info here, maybe even a diagram.

.. identity-recognition-what-is-start

Identity recognition is the process of matching customer profile information that arrives through event streams with known customer profiles. Identity recognition relies on linking keys that correlate important identifiers in source data with customer identifiers in events. Identity recognition is the only way to associate anonymous customer activity with known profiles.

.. identity-recognition-what-is-end


.. _identity-recognition-linking-keys:

About linking keys
==================================================

.. identity-recognition-linking-keys-start

A linking key is an event type field that with unique customer identifiers. For example:

* Email addresses
* Phone numbers
* Loyalty IDs
* Customer IDs

All events with linking keys are compared to known profiles. When a linking key matches a value in a known profile the event associates with the known profile.

Linking keys are defined in priority order. When two or more linking keys match, the highest priority matching linking key determines the profile to which the event associates.

An event with no linking keys is still processed, but is not associated with a known profile.

.. identity-recognition-linking-keys-end


.. _identity-recognition-multi-key:

Multiple linking keys
--------------------------------------------------

.. identity-recognition-multi-key-start

It is possible for more than one linking key to be usable to associate events data with known customer profiles.

.. tip:: Configure more than one event type for event streams that can match to a linking key. This increases the opportunities to associate anonymous events with known customer profiles.

For example, an event type for an event stream might define both **email** and **loyalty_id** as linking keys. If an event only has **loyalty_id**, that value is used for identity recognition. If an event arrives later on from the same session carries **email** *and* **loyalty_id**, both values have a chance to match values in known profiles.

.. important:: Event types for an event stream should include all identifiers that can help match anonymous events to known customer profiles. This is true even if sometimes the event type is not present in the event schema. Use **coalesce** in the expression for identifiers with variable names:

   .. code-block:: sql

      coalesce(loyalty_id, member_id, rewards_number)

   This approach maps any of the declared field names to a single **loyalty_id** linking key.

.. identity-recognition-multi-key-end


.. _identity-recognition-quality:

Identifier quality
--------------------------------------------------

.. identity-recognition-quality-start

Some identifiers are better than others. The quality of identifiers used as linking keys directly affects the likelihood of matching customer profile data that arrives through an event stream with a known customer profile.

There are two types of identifiers:

* :ref:`Strong <identity-recognition-quality-strong>`
* :ref:`Weak <identity-recognition-quality-weak>`

.. note:: The best linking keys are *also* configured as Stitch rules in the batched identity resolution process. When a streamed event has the same email address, loyalty ID, or customer ID that Amperity uses for identity resolution the event is associated with a unified customer profile.

   This ensures behavioral signals from event streams enrich known customer profiles and that batched data sources accurately describe purchase histories, loyalty tiers, and demographic attributes.

.. identity-recognition-quality-end


.. _identity-recognition-quality-strong:

Strong linking keys
++++++++++++++++++++++++++++++++++++++++++++++++++

.. identity-recognition-quality-strong-start

Strong identifiers are globally unique, stable over time, and are shared across data sources:

.. list-table::
   :widths: 40 60
   :header-rows: 0

   * - **Email address**
     - An email address is the most portable cross-channel identifier. This value should be assigned to a foreign key *and* be assigned a position in the Stitch rules hierarchy. Any event with an email address is highly likely to match to a known customer profile.

   * - **Phone number**
     - A phone number is a strong idenfier for SMS-driven, food delivery, or other platforms where email is not the primary contact point. Any event with a phone number is highly likely to match to a known customer profile.

       .. note:: All phone numbers should be in E.164 format for consistent matching to customer profiles.

   * - **Loyalty account identifiers**
     - A unique identifier for a loyalty program. This value should be assigned to a foreign key *and* be assigned a position in the Stitch rules hierarchy. Any event with a loyalty account identifier is highly likely to match to a known customer profile.

   * - **External customer identifiers**
     - Your brand's internal identifier for a customer. This is a strong identifier when the same value is used across multiple systems *and* the data from those systems is made available to Amperity.

.. identity-recognition-quality-strong-end


.. _identity-recognition-quality-weak:

Weak linking keys
++++++++++++++++++++++++++++++++++++++++++++++++++

.. identity-recognition-quality-weak-start

Weak identifiers are useful within a single session or platform but do not connect to the broader identity graph:

.. list-table::
   :widths: 40 60
   :header-rows: 0

   * - **Cookie and device identifiers**
     - Anonymous, per-device, and reset when a browser or app is cleared. These are useful for building a pre-login behavioral trail, but are not as useful for cross-channel matching.

       Configure cookie and device identifiers as a linking key only when anonymous-to-known signals have other supporting event identifiers with stronger signals.

   * - **Session identifiers**
     - Session IDs are ephemeral and are single-use.

       .. warning:: Do not use session identifiers as linking keys.

   * - **Platform-internal record identifiers**
     - Order IDs, ticket IDs, delivery IDs identify a transaction, not a customer. Platform-internal identifiers are not suitable as linking keys in most cases.

.. identity-recognition-quality-weak-end


.. _identity-recognition-outcomes:

About recognition outcomes
==================================================

.. identity-recognition-outcomes-start

For each incoming event, identity resolution produces one of four outcomes:

* :ref:`Merge <identity-recognition-outcome-merge>`
* :ref:`Passthrough <identity-recognition-outcome-passthrough>`
* :ref:`Unclaimed <identity-recognition-outcome-unclaimed>`
* :ref:`Unlinked <identity-recognition-outcome-unlinked>`

.. identity-recognition-outcomes-end


.. _identity-recognition-outcome-merge:

Merge
--------------------------------------------------

.. identity-recognition-outcome-merge-start

A linking key value matches profiles from two or more groups, at least one of which is an anonymous profile.

One or more anonymous profile merge into known profile, including the full event history of the anonymous profile.

A **Merge** outcome resolves an anonymous-to-known profile.

.. identity-recognition-outcome-merge-end

.. identity-recognition-outcome-merge-note-start

.. note:: Anonymous profiles are absorbed by known profiles after a **Merge** outcome. Any events attributed to the anonymous profile are re-attributed to the known profile.

.. identity-recognition-outcome-merge-note-end

.. identity-recognition-outcome-merge-example-start

.. admonition:: Example of merge outcomes

  Record F arrives through the event stream and matches a linking key in a known profile. Record F matches records D and E in an anonymous profile. Records D and E were previously in an :ref:`unlinked anonymous profile <identity-recognition-outcome-unlinked>` A merge outcome merges all matching records, including the reconrds in the previously anonymous profile into a known profile.

   .. image:: ../../images/identity-recognition-merge.png

.. _identity-recognition-outcome-passthrough:

Passthrough
--------------------------------------------------

.. identity-recognition-outcome-passthrough-start

A linking key value matches one or more existing known profiles. The event associates with the highest-priority known profile. An anonymous profile is not created. A **Merge** outcome does not occur. **Passthrough** is the most common outcome for returning customers.

.. identity-recognition-outcome-passthrough-end

.. identity-recognition-outcome-passthrough-example-start

.. admonition:: Example of passthrough outcomes

   Record A arrives through the event stream and matches a linking key in a known profile. The event is immediately added to the esisting known profile.

   .. image:: ../../images/identity-recognition-passthrough.png
      :width: 500 px
      :alt: A passthrough outcome for identity recognition of events in an event stream
      :align: left
      :class: no-scaled-link

.. identity-recognition-outcome-passthrough-example-end


.. _identity-recognition-outcome-unclaimed:

Unclaimed
--------------------------------------------------

.. identity-recognition-outcome-unclaimed-start

A linking key value matches a known profile, but the specific key-value pair has not been recorded for the matching profile.

A new linking key is added to the known profile and associated with the event. For example, a profile previously linked by email, but now links also with loyalty ID when loyalty events arrive.

.. identity-recognition-outcome-unclaimed-end

.. identity-recognition-outcome-unclaimed-example-start

.. admonition:: Example of unclaimed outcomes

   Event B arrives through the event stream. Event B matches record 1 on email address, but Event B *also* arrives with a loyalty ID. A loyalty ID is a strong identifier for identity recognition. An unclaimed outcome adds the loyalty ID to the linking keys for the known profile.

   .. image:: ../../images/identity-recognition-unclaimed1.png
      :width: 500 px
      :alt: An unclaimed outcome for identity recognition of events in an event stream.
      :align: left
      :class: no-scaled-link

   Event C arrives through the event stream and matches on the linking key for loyalty ID. A passthrough outcome adds event C to the known profile.

   .. image:: ../../images/identity-recognition-unclaimed2.png
      :width: 500 px
      :alt: An unclaimed outcome for identity recognition of events in an event stream.
      :align: left
      :class: no-scaled-link

.. identity-recognition-outcome-unclaimed-example-end


.. _identity-recognition-outcome-unlinked:

Unlinked
--------------------------------------------------

.. identity-recognition-outcome-unlinked-start

A linking key value does not match any existing profile. A new anonymous profile is linked to the identifier and to the associated event. Future events with this identifier will match to the anonymous profile until a **Merge** outcome associates it with a known profile.

.. identity-recognition-outcome-unlinked-end

.. identity-recognition-outcome-unlinked-example-start

.. admonition:: Example of unlinked outcomes

   Record D arrives through the event stream and does not match any known profiles. An unlinked outcome creates an anonymous profile with only record D.

   .. image:: ../../images/identity-recognition-unlinked1.png
      :width: 500 px
      :alt: An unlinked outcome for identity recognition of events in an event stream.
      :align: left
      :class: no-scaled-link

   Record E arrives through the event stream and does not match any known profiles. Record E matches record D on an anonymous identifier. An unlinked outcome adds record E to the anonymous profile with record D.

   .. image:: ../../images/identity-recognition-unlinked2.png
      :width: 500 px
      :alt: An unlinked outcome for identity recognition of events in an event stream.
      :align: left
      :class: no-scaled-link

   Record F arrives through the event stream and matches a linking key in a known profile. A :ref:`merge outcome <identity-recognition-outcome-merge>` merges the anonymous records to a known profile.

.. _identity-recognition-anonymous-profiles:

About anonymous profiles
==================================================

.. identity-recognition-anonymous-profiles-start

An anonymous profile is created automatically when an event arrives with a linking key that does not match any known profile.

An anonymous profile is treated like a customer profile. It accumulates attribute updates and belongs to segments just like known profiles. The only difference is that anonymous profiles have no association with a known profile created by batched identity resolution. Anonymous profiles exists only within the context of events data.

Anonymous profiles are useful when they can be associated with known profiles.

.. identity-recognition-anonymous-profiles-end


.. _identity-recognition-anonymous-to-known:

Anonymous-to-known resolution
--------------------------------------------------

.. identity-recognition-anonymous-to-known-start

The merge from an anonymous profile to a known profile happens automatically, in real time, at the moment an event arrives that connects the two identities.

A typical sequence:

.. list-table::
   :widths: 10 90
   :header-rows: 0

   * - .. image:: ../../images/steps-01.png
          :width: 60 px
          :alt: Step one.
          :align: center
          :class: no-scaled-link
     - A visitor lands on your website. A behavioral event is sent with a cookie ID (**user_pseudo_id**) as the linking key. No known profile matches, so Amperity creates an anonymous profile linked to the cookie ID.

   * - .. image:: ../../images/steps-02.png
          :width: 60 px
          :alt: Step two.
          :align: center
          :class: no-scaled-link
     - The visitor browses, adds items to a cart, and abandons. Each event is associated with the anonymous profile. Aggregation attributes accumulate.

   * - .. image:: ../../images/steps-03.png
          :width: 60 px
          :alt: Step three.
          :align: center
          :class: no-scaled-link
     - The visitor returns and signs in. A sign-in event is sent carrying both the cookie ID *and* their email address.

   * - .. image:: ../../images/steps-04.png
          :width: 60 px
          :alt: Step four.
          :align: center
          :class: no-scaled-link
     - Amperity resolves the email to a known profile. It detects that the cookie ID is linked to the anonymous profile. A merge is triggered: the anonymous profile's event history and accumulated attributes are folded into the known profile.

   * - .. image:: ../../images/steps-05.png
          :width: 60 px
          :alt: Step five.
          :align: center
          :class: no-scaled-link
     - Segment membership for the known profile is immediately re-evaluated with the full picture, including all the anonymous session's events. If the merged data causes the profile to enter a journey's inclusion segment, the journey fires.


The entire resolution and merge happens within the same pipeline tick as the sign-in event. No batch process or scheduled job is required.

.. identity-recognition-anonymous-to-known-end


.. _identity-recognition-enables-activation:

Activate profiles in real-time
==================================================

.. identity-recognition-enables-activation-start

Identity recognition is an important step toward taking action in real-time as customers interact with your brand.

Recognition is successful when the outcome is **Merge**, **Passthrough**, or **Unclaimed**. The event is associated with a known profile. Customer attributes are updated and segment membership is reevaluated, after which the recognized customer profile can take any of the following paths:

#. Enter a just-in-time workflow whose inclusion segment matches the updated known state. This creates a point activation in a journey that sends profile data to a configured destination or to a configured webhook. For example, to an email platform or an SMS provider for follow-up messaging, to an online advertising or paid media network, or to programmatic custom webhooks to support custom workflows.
#. Be blocked from activation by a suppression segment.
#. Be queried by external programmatic workflows using the Profile API v3 endpoints for custom real-time workflows using your own application logic.

All of these paths happen in real-time. A single event can take all three activation paths simultaneously, depending on how segments and destinations are configured. There is no limit to the number of just-in-time workflows that can respond to a single resolved event.

.. identity-recognition-enables-activation-end

.. identity-recognition-enables-activation-note-start

.. note:: An **Unlinked** outcome creates an anonymous profile that accumulates event history immediately. An anonymous profile cannot be sent to just-in-time workflows that require known profile attributes, but can be sent to just-in-time workflows designed for anonymous visitors.

   When an anonymous profile is merged into a known profile, the full event history transfers to the known profile and segment membership is reevaluated.

An event without identifiable linking keys is accepted and queued, but produces no actionable outcome. No profiles are updated. Segments are not reevaluated. Nothing is sent to a just-in-time workflow. The profile is not available for marketing use cases.

.. identity-recognition-enables-activation-note-end
